Unified Architecture Overview & Technical Audit

B.I.N.D — Kiến Trúc Toàn Diện

Hệ thống quản lý tri thức cá nhân (Personal Knowledge Management) chạy cục bộ (Local-first) trên Windows desktop. Bản báo cáo này hợp nhất toàn bộ cấu trúc kiến trúc hệ thống post-migration (React + WebView2), sơ đồ AI local/cloud, và giải pháp chuyển đổi tài liệu hoàn toàn offline chất lượng cao (Pandoc & IBM Docling) giúp tối ưu hóa nguồn dữ liệu đầu vào cho RAG và Đồ thị liên kết tri thức.

01 Kiến Trúc Phân Lớp & Công Nghệ Core

B.I.N.D được thiết kế trên mô hình 3 lớp tách biệt chạy trong một tiến trình duy nhất trên Windows. Lớp giao diện (React 19) giao tiếp trực tiếp với lớp C# Backend thông qua một Bridge IPC nội bộ độ trễ thấp, kết hợp với các MCP Agent ngoại vi để điều khiển Revit 2025.

Host Shell (WPF)

BIND.App

Window host mỏng bằng C# WPF, sử dụng WebView2 toàn màn hình mà không cần XAML UI phức tạp. Quản lý vòng đời tiến trình, Dependency Injection (Microsoft.Extensions.Hosting) và Serilog ghi log local.

UI Layer (SPA)

frontend/ (React 19 + Vite)

Ứng dụng React chạy độc lập, render qua WebView2 và được phục vụ từ virtual host bind.app.local hoàn toàn offline. Đồ thị tri thức sử dụng D3-force Canvas để tối ưu hóa hiệu năng render.

Bridge IPC

BIND.Bridge

Hệ thống truyền tin hai chiều qua postMessage và PostWebMessageAsJson. Định tuyến các handler theo prefix (chat, graph, agent, mcp, config, ingest, entities...) giúp đảm bảo luồng điều khiển đồng bộ.

Domain Core

BIND.Core

Trọng tâm lưu trữ các định nghĩa nghiệp vụ, Models và Interfaces (như ILlmProvider, IExtractor, IEmbedder) hoàn toàn độc lập, không phụ thuộc vào framework bên ngoài.

Data Store

BIND.Data

Sử dụng cơ sở dữ liệu SQLite thông qua Dapper, kết hợp extension sqlite-vec để lưu trữ và tìm kiếm vector nhúng (vector dimension 1024). Hỗ trợ cơ chế tự động chạy 4 migrations dữ liệu.

Ingestion Pipeline

BIND.Ingestion

Hệ thống nạp dữ liệu song song sử dụng Channel<T>. Tích hợp bộ chuyển đổi tài liệu cấu trúc (Pandoc cho Word, Docling cho PDF, OpenMcdf cho Revit, Tesseract cho OCR và Florence-2 ONNX cho chú thích ảnh).

AI & RAG Engine

BIND.AI

Quản lý các mô hình ngôn ngữ Cloud (Gemini, Claude, GPT) kết hợp mô hình nhúng cục bộ (BGE-M3 GGUF chạy trên CPU qua LLamaSharp). Cung cấp cơ chế RAG, Auto-tag và MCP Agent.

IPC & AddIn

BIND.IPC & Revit.AddIn

Kết nối trực tiếp tới Revit 2025 qua Named Pipe \\.\pipe\bind-revit. Sử dụng cơ chế ExternalEvent để thực thi các lệnh mô hình hóa BIM trên main thread của Revit một cách an toàn.

MCP Revit Agent

THBIM-MCP-Server (Node)

Subprocess Node.js khởi chạy tự động từ config, kết nối qua StdIO MCP C#. Cấp hơn 90 công cụ chuyên biệt để điều khiển, truy vấn và tạo lập đối tượng BIM trong Revit thông qua cổng HTTP 5050 bridge.

04 Quy Trình Nạp Dữ Liệu & Trích Xuất Tài Liệu Offline

Một hệ thống RAG tốt bắt buộc phải có nguồn dữ liệu đầu vào chất lượng cao. B.I.N.D đã loại bỏ các heuristics thô sơ dễ gây vỡ cấu trúc bảng biểu và danh sách của PDF/DOCX, thay thế hoàn toàn bằng **Pandoc CLI** và **IBM Docling MCP** chạy offline trên thiết bị.

Hình 2. Quy trình xử lý tài liệu đa luồng offline của B.I.N.D

4.1 Tải & Chuyển Đổi Word (.docx) Bằng Pandoc

Bản Windows di động của pandoc.exe (~40MB) được đóng gói trực tiếp vào thư mục assets/tools/. C# WordTextExtractor.cs gọi trực tiếp CLI để sinh ra định dạng GitHub Flavored Markdown (GFM) trong chưa đầy 50ms. 

// Thực thi CLI chuyển đổi offline độ chính xác tuyệt đối
var startInfo = new ProcessStartInfo
{
    FileName = pandocPath,
    Arguments = $"\"{filePath}\" -t gfm --wrap=none -o \"{tempOutFile}\"",
    UseShellExecute = false,
    CreateNoWindow = true
};

4.2 Tích Hợp IBM Docling Trích Xuất Bảng Biểu PDF Phức Tạp

PDF chứa các bảng biểu kỹ thuật phức tạp (như thuyết minh kỹ thuật BIM, bảng danh mục vật liệu có ô gộp) sẽ được gửi đến IBM Docling thông qua Node.js MCP Server. Docling sử dụng các mô hình Layout-Analysis và Table-Transformer chạy local để trả về cấu trúc bảng Markdown hoàn chỉnh, loại bỏ nhiễu tiêu đề trang (Header) và chân trang (Footer) trước khi chuyển tiếp vào RAG.

05 Cơ Chế Tìm Kiếm RAG & Cập Nhật Đồ Thị Tri Thức

Để giải quyết triệt để vấn đề kết quả RAG thiếu chính xác và thỉnh thoảng biến mất node liên kết trên Đồ thị tri thức, B.I.N.D triển khai mô hình **Tìm kiếm lai (Hybrid Search)** kết hợp **Reranker offline** và cơ chế **Gom nhóm token (Token Coalescing)** ở tầng Bridge để loại bỏ hoàn toàn hiện tượng lag giao diện.

Hình 3. Cơ chế tìm kiếm lai (Hybrid RAG) và luồng gom nhóm token (Token Coalescing)

5.1 Tại Sao Node Liên Kết Đồ Thị Bị Biến Mất?

Trong hệ thống cũ, mối liên kết cạnh đồ thị được tính toán offline thông qua SimilarityIndexer, ghi vào bảng GraphConnections(SourceId, TargetId, Weight) bằng cách tính Cosine giữa centroid các chunk-embedding của hai Entity.

Giải thích hiện tượng lỗi thực tế: Khi tài liệu bị trích xuất heuristics sai cách (làm mất từ khóa hoặc gộp sai chủ đề), centroid vector của Entity sẽ bị lệch đáng kể. Khi đó, Weight liên kết rơi xuống dưới ngưỡng lọc cạnh đồ thị (similarity threshold) khiến node bị cô lập hoàn toàn và biến mất trên giao diện D3 Canvas. Việc áp dụng Pandoc và Docling sẽ chuẩn hóa chất lượng Markdown thô, trực tiếp khôi phục lại tính chính xác cho đồ thị.

5.2 Cơ Chế Gom Nhóm Token (Token Coalescing) Chống Lag UI

Khi stream phản hồi từ các mô hình cực nhanh như Gemini 2.5 Flash, lượng tin nhắn chat:token bắn liên tục qua IPC Bridge có thể đạt tần suất >200 sự kiện/giây. Việc này kích hoạt React liên tục render lại (re-render) màn hình, gây hiện tượng nghẽn luồng UI. B.I.N.D khắc phục bằng một bộ đệm gom nhóm (coalescing buffer) tại C# Host Bridge: tích lũy các token delta nhận được trong 30-50ms và chỉ phát một gói gộp duy nhất sang WebView2, đảm bảo UI hoạt động mượt mà ở mức 60fps.

06 Thông Số Thuật Toán Thực Tế (Ground Truth)

Các thông số cấu hình và tham số thuật toán cơ bản đang chạy trong lõi backend B.I.N.D.

6.1 Cấu Hình Chunking (MarkdownChunker)

  • Mục tiêu độ dài (Target): ~500 tokens (xấp xỉ 2000 ký tự).
  • Độ dài tối đa (Max Budget): 2000 tokens per chunk.
  • Độ chồng lấp (Overlap): ~50 tokens, chuyển tiếp tiền tố đuôi.
  • Nguyên tắc biên giới (Boundaries): Cố định giữ chung tiêu đề (Heading) và đoạn văn đầu tiên. Code blocks là nguyên tử (không cắt ở giữa fence). Đoạn văn thường được chia cắt theo dấu câu tiếng Việt dựa trên Regular Expression.

6.2 Truy Vấn Vector (sqlite-vec)

  • Bảng ảo (Virtual Table): ChunkEmbeddings USING vec0(chunk_id, embedding FLOAT[1024]).
  • Thuật toán khoảng cách: L2 (Euclidean distance). Do vector của BGE-M3 được chuẩn hóa đơn vị (unit-normalized) đầu vào, khoảng cách L2 được chuyển đổi chính xác sang Cosine Similarity qua công thức: Score = 1 − d²/2.
  • Lọc điểm tương đồng: Lọc cứng các chunk có Cosine Score < 0.5.

6.3 Định Tuyến Công Cụ MCP (MCP Tool Retrieval)

  • Số lượng phân phối: Tối đa 12 công cụ/lượt gửi đến LLM.
  • Thuật toán chấm điểm: Cosine(BGE-M3(query), BGE-M3("name: description")) + lexical boost (cộng thêm 0.20 nếu trùng khớp chính xác tên, cộng thêm 0.10 cho mỗi token khớp có độ dài >3 ký tự).
  • Bảo vệ (Denylist): Chặn cứng các công cụ nhạy cảm hệ thống.

6.4 Vòng Lặp Revit Agent (Revit Agent Loop)

  • Giới hạn vòng lặp (Iteration Cap): Tối đa 6 vòng lặp phản hồi/câu hỏi.
  • Nhiệt độ (Temperature): Khóa cứng mức 0.2 để giảm tính ngẫu nhiên, tăng tính chính xác khi gọi tool.
  • Cơ chế xác thực an toàn: Các công cụ thay đổi hoặc xóa mô hình BIM bắt buộc phải phản hồi thẻ xác nhận (Confirm Card) trên giao diện để người dùng nhấn nút duyệt, không có cơ chế tự động duyệt hàng loạt.

08 Bản Đồ Cải Tiến & Quản Trị Rủi Ro Kỹ Thuật

Lộ trình thực thi nâng cấp hệ thống được xếp hạng theo tỷ lệ ROI (Tỷ suất hiệu quả thực tế trên công sức thực hiện).

Rủi ro 1: Rò rỉ thông tin Cloud LLM

Việc sử dụng các cloud provider như Gemini hay Anthropic đồng nghĩa với việc gửi các đoạn cắt ngữ cảnh tài liệu lên cloud. Giải pháp giảm thiểu: Triển khai bộ lọc dữ liệu nhạy cảm ở phía client, cấu hình API key độc lập theo từng dự án và cam kết không gửi nội dung file thô lên cloud — chỉ gửi các đoạn cắt nhỏ đã được vector hóa.

Rủi ro 2: Chi phí API cloud tăng vọt

Sử dụng các model thông minh như Claude Sonnet 4.5 Thinking kết hợp với RAG context dài có thể ngốn 10K-20K tokens/câu hỏi. Giải pháp giảm thiểu: Thiết lập cơ chế kiểm soát token cap trên giao diện người dùng và bộ ước lượng giá tiền real-time trước khi phát lệnh chat.

Rủi ro 3: Plaintext API Keys trên file cấu hình

Tệp cấu hình bind_config.json lưu trữ API keys dạng văn bản thô dễ bị đánh cắp nếu máy bị nhiễm malware. Giải pháp giảm thiểu: Khóa tệp tin cấu hình bằng cơ chế DPAPI (Data Protection API) của Windows để chỉ định đúng tài khoản người dùng Windows hiện tại mới có quyền đọc khóa bí mật.

Lợi thế lớn: Luôn duy trì offline-first

Toàn bộ các tác vụ nạp (Ingest), vector hóa (Embed), tìm kiếm cục bộ (Search) và vẽ đồ thị (Graph) vẫn chạy 100% offline không cần mạng. Khi mất kết nối Internet, hệ thống chỉ vô hiệu hóa nhẹ nhàng phần Chat LLM và phản hồi thông báo thân thiện tới người dùng.

Lộ trình thực hiện khuyến nghị (ROI giảm dần)

  1. Hạng mục 1: Token Coalescing 30-50ms (Làm ngay): Khắc phục tức thời lỗi đơ lag giao diện chat khi stream phản hồi. Công sức thực hiện: <1 ngày.
  2. Hạng mục 2: Tích hợp Pandoc CLI cho DOCX (Làm ngay): Tăng độ chính xác trích xuất Word lên 100% tuyệt đối. Công sức: <1 ngày.
  3. Hạng mục 3: Tích hợp IBM Docling qua MCP Server: Khôi phục hoàn chỉnh cấu trúc bảng PDF và loại bỏ nhiễu Header/Footer. Công sức: 1-2 ngày.
  4. Hạng mục 4: HWND Subclassing cho Frameless Window: Loại bỏ triệt để lỗi hụt kéo thả di chuyển cửa sổ và resize viền. Công sức: 1 ngày.
  5. Hạng mục 5: Tích hợp Cross-Encoder Reranker offline: Nâng chất lượng câu trả lời của RAG lên thêm 20-30% trên CPU máy khách. Công sức: 2 ngày.
  6. Hạng mục 6: Chuyển đổi giao diện Graph sang Sigma.js v3: Bảo đảm đồ thị hoạt động mượt mà khi cơ sở dữ liệu phình to. Công sức: 3 ngày.
